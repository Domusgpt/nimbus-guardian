Claude Code

Meet the Observability Plugin

Bring observability data directly into Claude Code. The plugin mirrors the dashboard APIs so every conversation can surface status, streaming, history, and incident context on demand.

Included Commands

  • /observability-status – live snapshot with sink health.
  • /observability-history – rolling window with summary insights.
  • /observability-stream – SSE stream viewer with graceful teardown.
  • /observability-incidents – filterable incident tracker.
  • /observability-digest – snapshot, history, and incident bundle.

Trust & Install

  1. Open Claude Code in the repository root.
  2. Add the local marketplace: /plugin marketplace add ./claude-plugins.
  3. Install the bundle: /plugin install nimbus-guardian-observability@claude-plugins.
  4. Run /observability-status to verify the commands are available.

Each command includes inline documentation for quick reference.

CLI Digest

One Command for Total Telemetry

Generate the same telemetry from your terminal or automation. Batching, JSON output, and header controls keep the CLI aligned with Claude Code and the dashboard.

Quick Start

npm install -g nimbus-guardian
nimbus observability-digest --history --incidents --format table

Use --format json for automation pipelines or --once for a single update.

Outputs You Get

  • Latest snapshot with fingerprint and sink summaries.
  • Rolling history including overview analytics.
  • Incident ledger with duration, status, and reason codes.
  • Exit codes wired for CI/CD fast-fails.
Orbit Terminal

Preview the Commands Before You Install

Review command syntax, copy common flags, and skim quick notes before running anything locally. The Orbit Terminal mirrors the Claude plugin docs so teams can share the same reference. 

nimbus observability-status \
  --timeout 8000 \
  --format table
nimbus observability-stream \
  --cookie "guardian_session=..." \
  --heartbeat 15000
nimbus observability-history \
  --limit 10 \
  --overview
nimbus observability-incidents \
  --status degraded,failing \
  --format json
nimbus observability-digest \
  --history --incidents \
  --output ./orbit-digest.json

Why Run Status?

Collect a one-shot snapshot with fingerprint deltas, sink summaries, and health badges. Ideal for CI checks or quick reviews inside Claude conversations.

Why Run Stream?

Follow live fingerprint updates. Graceful teardown keeps the terminal clean while sink metrics refresh as soon as exporters emit new telemetry.

Why Run History?

Review the rolling buffer, summary analytics, and incident counts in one view. Helpful for daily standups or retrospectives.

Why Run Incidents?

Filter for degraded or failing windows to see how long each sink struggled and why alerts fired.

Why Run Digest?

Bundle a snapshot, history insights, and the incident ledger into one JSON or table-friendly report for sharing or automation.

Operator FAQ

Answers at a Glance

Keep the rollout grounded with concise responses to the questions stakeholders ask most often. The FAQ keeps installation, telemetry, and roadmap expectations aligned with the README and docs plan.

How do we install the Claude plugin safely?

Trust the repository folder in Claude Code, then run /plugin marketplace add ./claude-plugins followed by /plugin install nimbus-guardian-observability@claude-plugins. The repository’s marketplace manifest pins the exact plugin version so reviews stay predictable.

Can the CLI share the same telemetry?

Yes. The nimbus observability-digest command calls the same exporter endpoints as the plugin and returns snapshots, history, and incident reports in one run. Pass --format json to integrate the output with automation or monitoring pipelines.

Where should we send stakeholders for roadmap updates?

Point them to parserstor.com for constellation announcements. The site highlights Nimbus today and cues the upcoming Reposiologist and Vib3-Scribe releases without over-promising timelines.

What if someone still prefers the dashboard?

The dashboard is still the primary destination. The plugin, CLI, and Orbit microsite exist to keep teams productive while the UI polish finishes. Everything here maps directly back to the dashboard APIs, so there is no divergent behavior to reconcile later.
